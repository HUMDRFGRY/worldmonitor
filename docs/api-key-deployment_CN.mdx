## 概述

桌面端云回退依赖 `WORLDMONITOR_API_KEY`。没有有效 key 时，桌面应用会以本地模式（sidecar）运行。注册表单会收集邮箱并写入 Convex DB，以便后续发放 key。

## 架构

```
Desktop App                          Cloud (Vercel)
┌──────────────────┐                ┌──────────────────────┐
│ fetch('/api/...')│                │ api/[domain]/v1/[rpc]│
│        │         │                │        │              │
│ ┌──────▼───────┐ │                │ ┌──────▼───────┐      │
│ │ sidecar try  │ │                │ │ validateApiKey│      │
│ │ (local-first)│ │                │ │ (origin-aware)│      │
│ └──────┬───────┘ │                │ └──────┬───────┘      │
│   fail │         │                │   401 if invalid      │
│ ┌──────▼───────┐ │   fallback    │                       │
│ │ WM key check │─┼──────────────►│ ┌──────────────┐      │
│ │ (gate)       │ │  +header      │ │ route handler │      │
│ └──────────────┘ │               │ └──────────────┘      │
└──────────────────┘               └──────────────────────┘
```

## 必需环境变量

### Vercel

| 变量 | 描述 | 示例 |
|----------|-------------|---------|
| `WORLDMONITOR_VALID_KEYS` | 以逗号分隔的有效 API key 列表 | `wm_abc123def456,wm_xyz789` |
| `CONVEX_URL` | Convex 部署 URL（来自 `npx convex deploy`） | `https://xyz-123.convex.cloud` |

### 生成 API key

Key 长度必须至少 16 个字符（客户端会校验）。推荐格式：

```bash
# 生成一个 key
openssl rand -hex 24 | sed 's/^/wm_/'
# 示例输出: wm_a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6
```

将其添加到 Vercel 控制台中的 `WORLDMONITOR_VALID_KEYS`，用逗号分隔，不要加空格。

## Convex 设置

### 首次部署

```bash
# 1. 安装（已在 package.json 中声明）
npm install

# 2. 登录 Convex
npx convex login

# 3. 初始化项目（会创建带有 CONVEX_URL 的 .env.local）
npx convex init

# 4. 部署 schema 和 functions
npx convex deploy

# 5. 把部署 URL 复制到 Vercel 环境变量
# 该 URL 会被 `npx convex deploy` 打印出来，并保存到 .env.local
```

### 验证 Convex 部署

```bash
# 类型检查 Convex functions
npx convex dev --typecheck

# 打开 Convex dashboard 查看注册记录
npx convex dashboard
```

### Schema

`registrations` 表会存储：

| 字段 | 类型 | 描述 |
|-------|------|-------------|
| `email` | string | 原始邮箱（用于显示） |
| `normalizedEmail` | string | 转小写后的邮箱（用于去重） |
| `registeredAt` | number | Unix 时间戳 |
| `source` | string? | 注册来源 |
| `appVersion` | string? | 桌面应用版本 |

通过 `normalizedEmail` 建索引，以便检测重复。

## 安全模型

### 客户端（桌面应用）

- `installRuntimeFetchPatch()` 会在允许云回退前检查 `WORLDMONITOR_API_KEY`
- key 必须存在且有效（至少 16 字符）
- `secretsReady` promise 会确保在第一次 fetch 前已经加载机密（2s 超时）
- fail-closed：任何 key 检查错误都会阻止云回退

### 服务端（Vercel edge）

- `api/_api-key.js` 会在 sebuf 路由上验证 `X-WorldMonitor-Key` header
- **基于来源**：桌面来源（`tauri.localhost`、`tauri://`、`asset://`）需要 key
- Web 来源（`worldmonitor.app`）可以不带 key 直接通过
- 非桌面来源即使带了 key header，也会继续校验 key
- 无效 key 会返回 `401 { error: "Invalid API key" }`

### CORS

`X-WorldMonitor-Key` 在 `server/cors.ts` 和 `api/_cors.js` 中都被允许。

## 验证清单

部署后：

- [ ] 在 Vercel 上设置 `WORLDMONITOR_VALID_KEYS`
- [ ] 在 Vercel 上设置 `CONVEX_URL`
- [ ] 运行 `npx convex deploy` 推送 schema
- [ ] 桌面端无 key：云回退被阻止（控制台显示 `cloud fallback blocked`）
- [ ] 桌面端带无效 key：sebuf 请求返回 `401`
- [ ] 桌面端带有效 key：云回退按原样工作
- [ ] Web 访问：不需要 key，正常工作
- [ ] 注册表单：提交邮箱，检查 Convex dashboard
- [ ] 重复邮箱：显示 "already registered"
- [ ] 现有设置页签（LLMs、API Keys、Debug）保持不变
