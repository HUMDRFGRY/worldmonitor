## `/api/health`

主健康检查端点。它会在一次 pipeline 调用中检查所有 Redis-backed 数据 key 和 seed 新鲜度元数据。

**认证：** 不需要。对所有 origin 开放（`Access-Control-Allow-Origin: *`）。

**HTTP 方法：** `GET`

### 查询参数

| 参数 | 取值 | 描述 |
|-----------|--------|-------------|
| `compact` | `1` | 省略每个 key 的详细信息；只返回有问题的 key |

### 响应状态码

| HTTP 状态 | 总体状态 | 含义 |
|-------------|---------------|---------|
| `200` | `HEALTHY` | 所有检查正常，没有警告 |
| `200` | `WARNING` | 有些 key 过旧，或按需 key 为空，但没有关键故障 |
| `503` | `DEGRADED` | 1-3 个 bootstrap key 为空 |
| `503` | `UNHEALTHY` | 4+ 个 bootstrap key 为空 |
| `503` | `REDIS_DOWN` | 无法连接 Redis |

### 响应体

```json
{
  "status": "HEALTHY | WARNING | DEGRADED | UNHEALTHY | REDIS_DOWN",
  "summary": {
    "total": 57,
    "ok": 52,
    "warn": 5,
    "crit": 0
  },
  "checkedAt": "2026-03-11T14:00:00.000Z",
  "checks": {
    "earthquakes": {
      "status": "OK",
      "records": 142,
      "seedAgeMin": 8,
      "maxStaleMin": 30
    }
  }
}
```

使用 `?compact=1` 时，`checks` 对象会被 `problems` 替代，后者只包含非 OK 的 key。

### Key 分类

Key 被分成三层，用于决定告警严重度：

| 层级 | 为空时的严重度 | 描述 |
|------|-------------------|-------------|
| **Bootstrap** | CRIT | 启动时必须存在的 seeded 数据。为空表示仪表盘缺少关键数据 |
| **Standalone** | CRIT（seeded）/ WARN（on-demand） | 由 seed 循环或 RPC handler 填充。按需 key 在首次请求前为空是预期行为 |
| **On-demand** | WARN | 由 RPC 调用懒加载填充。如果没人请求过数据，空是正常的 |

### 每个 Key 的状态

| 状态 | 严重度 | 含义 |
|--------|---------|------|
| `OK` | 绿色 | 数据存在，seed 新鲜 |
| `OK_CASCADE` | 绿色 | key 为空，但级联组里的兄弟 key 有数据（例如战区态势回退链） |
| `STALE_SEED` | 警告 | 数据存在，但 `seed-meta` 的年龄超过了 `maxStaleMin` |
| `EMPTY_ON_DEMAND` | 警告 | 按需 key 目前还没有数据 |
| `EMPTY` | 严重 | Bootstrap 或 standalone key 没有数据 |
| `EMPTY_DATA` | 严重 | key 存在，但记录数为 0 |

### 级联组

有些 key 使用回退链。如果任何兄弟节点有数据，空的兄弟会报告为 `OK_CASCADE`：

- **Theater Posture:** `theaterPostureLive` -> `theaterPosture`（stale） -> `theaterPostureBackup`
- **Military Flights:** `militaryFlights` -> `militaryFlightsStale`

### 陈旧阈值（maxStaleMin）

来自 `SEED_META` 的部分阈值：

| 领域 | 最大过期时间（分钟） | 说明 |
|--------|----------------|------|
| 市场报价、加密货币、板块 | 30 | 高频 relay 循环 |
| 地震、动荡、洞察 | 30 | 关键事件数据 |
| 预测 | 15 | Polymarket，变化很快 |
| 军机追踪 | 15 | 接近实时追踪 |
| 航班延误（FAA） | 60 | 机场延误快照 |
| 野火、气候异常 | 120 | 变化较慢的自然事件 |
| 网络威胁 | 480 | APT 数据更新频率较低 |
| BIS 数据、World Bank、矿产 | 2880-10080 | 机构数据，按周/月更新 |

### 示例请求

```bash
# 完整健康检查
curl -s https://api.worldmonitor.app/api/health | jq .

# 精简版（只看问题）
curl -s "https://api.worldmonitor.app/api/health?compact=1" | jq .

# UptimeRobot / 监控：检查 HTTP 状态码
curl -o /dev/null -s -w "%{http_code}" https://api.worldmonitor.app/api/health
# 返回 200（healthy/warning）或 503（degraded/unhealthy）
```

## `/api/seed-health`

面向 seed 循环新鲜度的专用端点。它只检查 `seed-meta:*` key，不拉取实际数据负载。

**认证：** 需要有效 API key 或被允许的 origin。

**HTTP 方法：** `GET`

### 响应状态码

| HTTP 状态 | 总体状态 | 含义 |
|-------------|---------------|---------|
| `200` | `healthy` | 所有 seed 循环都按时上报 |
| `200` | `warning` | 有些 seed 过旧（年龄 > 间隔的 2 倍） |
| `200` | `degraded` | 有些 seed 完全缺失 |
| `401` | - | API key 无效或缺失 |
| `503` | - | Redis 不可用 |

### 响应体

```json
{
  "overall": "healthy | warning | degraded",
  "checkedAt": 1710158400000,
  "seeds": {
    "seismology:earthquakes": {
      "status": "ok",
      "fetchedAt": 1710158100000,
      "recordCount": 142,
      "sourceVersion": null,
      "ageMinutes": 5,
      "stale": false
    },
    "market:stocks": {
      "status": "stale",
      "fetchedAt": 1710150000000,
      "recordCount": 85,
      "sourceVersion": null,
      "ageMinutes": 140,
      "stale": true
    }
  }
}
```

### 陈旧逻辑

当 seed 的年龄超过**配置间隔的 2 倍**时，就会被视为陈旧。这样可以容纳 cron/relay 时间上的正常抖动。

| 领域 | 间隔（分钟） | 超过多久算陈旧 |
|--------|---------------|-------------------|
| 预测、军机追踪 | 8 | 16 |
| 市场报价、地震、动荡 | 15 | 30 |
| ETF 流向、稳定币、 chokepoints | 30 | 60 |
| 服务状态、支出、野火 | 60 | 120 |
| 运费、卫星 | 90-120 | 180-240 |
| GPS 干扰、流离失所 | 360 | 720 |
| 伊朗事件、UCDP | 210-5040 | 420-10080 |

### 示例请求

```bash
curl -s https://api.worldmonitor.app/api/seed-health \
  -H "Origin: https://worldmonitor.app" | jq .
```

## 与监控工具集成

### UptimeRobot

把 `/api/health` 作为监控 URL。UptimeRobot 会检查 HTTP 状态：

- `200` = up
- `503` = down

如果要做关键字监控，可以检查响应体中的 `"status":"HEALTHY"`。

### 自定义告警

解析 JSON 响应来构建更细粒度的告警：

```bash
# 对任何关键 key 告警
STATUS=$(curl -s "https://api.worldmonitor.app/api/health?compact=1")
CRIT=$(echo "$STATUS" | jq '.summary.crit')
if [ "$CRIT" -gt 0 ]; then
  echo "CRITICAL: $CRIT data keys empty"
  echo "$STATUS" | jq '.problems'
fi
```

## 两个端点的区别

| 维度 | `/api/health` | `/api/seed-health` |
|--------|--------------|-------------------|
| **范围** | 数据 key + seed 元数据 | 仅 seed 元数据 |
| **认证** | 无（公开） | API key 或允许的 origin |
| **获取的数据** | 完整 Redis 值（用于统计记录数） | 仅 `seed-meta:*` key |
| **HTTP 503** | 有（DEGRADED/UNHEALTHY） | 没有（除非 Redis 挂了，否则始终 200） |
| **最适合** | 运行时监控、仪表盘健康检查 | 调试 seed 循环问题 |
| **响应大小** | 更大（57+ 个 key 带记录数） | 更小（42 个 seed 域） |
